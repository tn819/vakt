# Team Tier: Shared Config Registry, Centralized Secrets, Onboarding

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Design and implement the three team-tier forcing functions that transform mcpctl from a personal CLI tool into a shared-state coordination tool for engineering teams: (1) a shared org config registry, (2) centralized secrets management with vault-based access, and (3) a zero-friction onboarding flow.

**Architecture:** The team tier adds a server-side component (the org registry) that developer machines sync from. The registry is the source of truth; individual `~/.agents/mcp-config.json` files become derived outputs. Secrets are stored centrally with per-user/per-role access control. The CLI gains a `join` command for onboarding. A minimal web dashboard provides team visibility.

**Tech Stack:** TypeScript (for the team server), existing Bash CLI extended with `join`/`publish`/`pull` commands, REST API for registry sync, existing `~/.agents/` schema extended with org layer.

**Why these three features:**

These are the forcing functions that make the team upgrade obvious:

1. **Shared config registry** — the moment two developers need the same MCP server, uncoordinated local configs diverge. The registry solves the coordination problem.
2. **Centralized secrets** — shared service account credentials (GitHub org token, internal API keys) cannot live in any one developer's keychain. They need a team vault with access revocation on offboarding.
3. **Onboarding** — the most acute developer pain articulated in research: "on a new machine, I have to manually recreate all my MCP configs." `mcpctl join org/team` replaces hours of setup with one command.

```typescript
// src/types/org-config.ts
export interface OrgServer {
  command: string
  args: string[]
  env?: Record<string, string>
  approved: boolean
  approvedBy?: string
  approvedAt?: string
  pinnedVersion?: string
}

export interface OrgPolicy {
  allowUnlistedServers: boolean
}

export interface OrgConfig {
  org: string
  version: number
  servers: Record<string, OrgServer>
  skills: string[]
  policy: OrgPolicy
}

export function validateOrgConfig(config: unknown): config is OrgConfig {
  if (!config || typeof config !== 'object') return false
  const c = config as Record<string, unknown>
  return (
    typeof c.org === 'string' &&
    typeof c.version === 'number' &&
    typeof c.servers === 'object' &&
    typeof c.policy === 'object'
  )
}
```

### Task 2: Design the registry sync protocol

**Files:**
- Create: `docs/schema/registry-sync.md`
- Create: `src/lib/registry-sync.ts`

**Step 1: Document the sync protocol**

```markdown
# Registry Sync Protocol

## Endpoints

GET  /api/v1/orgs/:org/config          — fetch org config (requires auth token)
POST /api/v1/orgs/:org/config          — publish org config (admin only)
GET  /api/v1/orgs/:org/secrets/:key    — fetch secret reference metadata (not value)
POST /api/v1/auth/token                — exchange invite token for session token

## Auth

All requests carry `Authorization: Bearer <session-token>` header.
Session tokens are stored in OS keychain under `mcpctl-session-<org>`.
Invite tokens are single-use, expire in 48h, generated by admin.

## Merge semantics

On `mcpctl pull`:
1. Fetch org config from registry
2. Merge with local overrides (local wins on conflict)
3. Write merged result to ~/.agents/mcp-config.json
4. Run mcpctl sync to propagate to all installed tools

Local overrides live in ~/.agents/local-overrides.json (gitignored equivalent).
```

```typescript
// src/lib/registry-sync.ts
export function mergeOrgConfig(orgConfig: Partial<OrgConfig>, local: Partial<OrgConfig>): OrgConfig {
  return {
    ...orgConfig,
    servers: {
      ...orgConfig.servers,
      ...local.servers  // local wins
    },
    skills: [...(orgConfig.skills ?? []), ...(local.skills ?? [])]
  } as OrgConfig
}
```

**Step 1: Write the join command flow**

```bash
# src/commands/join.sh
join() {
  local org_team="${1:-}"
  if [[ -z "$org_team" ]]; then
    echo "Usage: mcpctl join <org>/<team>"
    exit 1
  fi

  local org="${org_team%%/*}"
  local team="${org_team##*/}"

  echo ""
  echo "Joining org: $org, team: $team"
  echo ""

  # Step 1: Prompt for invite token
  read -rsp "  Paste your invite token: " invite_token
  echo ""

  # Step 2: Exchange invite token for session token
  local session_token
  session_token=$(exchange_invite_token "$org" "$invite_token")

  # Step 3: Store session token in keychain
  store_secret "mcpctl-session-${org}" "$session_token"

  # Step 4: Pull org config
  mcpctl_pull "$org"

  # Step 5: Sync to all installed tools
  mcpctl_sync

  echo ""
  echo "  ✓ Joined $org/$team"
  echo "  ✓ $(count_servers) MCP servers configured"
  echo "  ✓ Synced to all installed AI coding tools"
  echo ""
}
```

### Task 4: Design the centralized secrets vault

**Files:**
- Create: `docs/schema/team-secrets.md`

**Step 1: Document the vault access model**

```markdown
# Team Secrets Vault

## Access model

Secrets in the team vault are named references, the same as individual secrets.
The difference: the secret VALUE lives server-side, resolved at sync time.

Individual secret (current): `secret:GITHUB_TOKEN` → resolved from local OS keychain
Team secret (new): `team-secret:GITHUB_ORG_TOKEN` → resolved from team vault, access-controlled

## Vault access tiers

| Tier | Who | What they can do |
|------|-----|-----------------|
| Owner | Org admin | Create/rotate/delete any secret |
| Manager | Team lead | Create secrets within their team scope |
| Member | Developer | Read (resolve) secrets they've been granted access to |

## Offboarding

When a member is removed from the org:
- Their session token is revoked (can no longer sync or resolve team secrets)
- Their personal keychain secrets are unaffected
- Team secrets they had access to are NOT automatically rotated (rotation is manual, for audit)
- Admin receives a "pending rotation recommendation" for all secrets the offboarded user had access to

## Secret rotation

`mcpctl secrets rotate team-secret:GITHUB_ORG_TOKEN <new-value>` — admin only
All member machines pick up the new value on next sync.
```

### Task 5: Design the team dashboard (web UI MVP)

**Files:**
- Create: `docs/schema/team-dashboard.md`

**Step 1: Define the minimum viable dashboard**

The team dashboard answers three questions for a Platform Engineering Lead:

1. **What's configured?** — list of MCP servers in the org registry, with approval status, version, and owner
2. **Where is it deployed?** — list of team members and which servers are active on their machines (last sync timestamp)
3. **What changed?** — config change log: who changed what, when, before/after diff

MVP screens (wireframe-level spec):

```
/dashboard
  ├── /servers         — org registry: list all approved servers, add/remove/approve
  ├── /members         — member list: join date, last sync, server coverage per member
  ├── /secrets         — vault: list named references (not values), manage access
  └── /activity        — change log: timestamp, actor, change description, diff link

/onboarding
  └── /invite          — generate single-use invite tokens for new members
```

**Step 2: Define the API surface the dashboard requires**

```
GET  /api/v1/orgs/:org/servers          — list all servers in registry
POST /api/v1/orgs/:org/servers          — add server (admin)
DELETE /api/v1/orgs/:org/servers/:name  — remove server (admin)
GET  /api/v1/orgs/:org/members          — list members with last-sync info
GET  /api/v1/orgs/:org/activity         — paginated change log
POST /api/v1/orgs/:org/invites          — generate invite token (admin)
DELETE /api/v1/orgs/:org/members/:id    — offboard member (admin)
```
